# Discord Integration Plan — KeyCoord

## Context

KeyCoord is a WoW Mythic+ key tracking system (Lambda + DynamoDB + React, deployed on AWS via CDK). Guild members submit their Mythic+ keystones via a WoW addon → website flow. The goal is to add Discord integration so guilds can:

1. Receive channel notifications when keys are added/updated
2. Run a `/keys` slash command to query current guild keys

Notifications are the higher-priority feature. Slash commands require a Discord Application with an interaction endpoint URL pointing at a Lambda.

---

## Setup UX

Guild setup happens entirely through Discord — no webhook URLs, no website form:

1. Guild member runs `/setup` in the Discord channel they want notifications in
2. Bot responds ephemerally: *"Click here to verify your WoW character: [Verify with Battle.net →]"* (link expires in 10 minutes)
3. User clicks → Blizzard OAuth on the website → callback verifies guild membership → saves config → bot posts "✅ Setup complete!" in the channel

The Blizzard OAuth step is unavoidable for membership verification but requires no manual data entry.

---

## Architecture

Both features are fully serverless. Discord's interaction model allows slash commands via HTTP POST to a Lambda — no persistent bot process required. Notifications are triggered by a DynamoDB Stream on the keys table, keeping `putKey.ts` free of Discord concerns. The bot posts to channels directly using its bot token (no webhook URLs).

**New infra:**
- 2 new DynamoDB tables (Discord configs; pending OAuth tokens)
- DynamoDB Stream enabled on the existing keys table
- 3 new Lambda handlers (`discordInteraction`, `discordNotification`, `discordSetup`)
- 2 new API Gateway routes
- 1 new website page (`/discord-auth`)

## DynamoDB Tables

### `key-coord-discord-config-table{suffix}`

| Attribute | Type | Role |
|---|---|---|
| `discordGuildId` | String | PK — Discord server snowflake ID |
| `sk` | String | Sort key — always literal `"CONFIG"` |
| `wowGuildId` | String | e.g. `us-area-52-example-guild` |
| `channelId` | String | Discord channel snowflake ID — where notifications are posted |
| `configuredBy` | String | battletag of configuring member |
| `configuredAt` | String | ISO-8601 timestamp |
| `notificationsEnabled` | Boolean | Toggle without deleting config |

**GSI:** `wowGuildId-index` (PK: `wowGuildId`) — projects `discordGuildId`, `channelId`, `notificationsEnabled`. Used by `discordNotification` to find the channel for a given WoW guild.

**`removalPolicy: RETAIN`** — holds user-configured data that should survive stack teardown.

### `key-coord-discord-pending-table{suffix}`

Stores short-lived tokens generated by the `/setup` command, consumed once by the OAuth callback.

| Attribute | Type | Role |
|---|---|---|
| `token` | String | PK — random UUID |
| `discordGuildId` | String | Discord server that initiated setup |
| `channelId` | String | Channel where `/setup` was run |
| `ttl` | Number | Epoch seconds — expires 10 minutes after creation |

**TTL enabled** on `ttl` attribute. No GSI needed — always looked up by token.

**`removalPolicy: DESTROY`** — ephemeral data, safe to drop.

## New Lambda Handlers

### `src/handlers/discordInteraction.ts`

**Route:** `POST /discord/interactions`

Registered in the Discord Developer Portal as the "Interactions Endpoint URL." Handles all slash commands.

**First action — Ed25519 signature verification** (must be before any business logic):

```typescript
const rawBody = event.isBase64Encoded
  ? Buffer.from(event.body ?? '', 'base64').toString('utf-8')
  : (event.body ?? '')

const valid = await verifyDiscordSignature(DISCORD_PUBLIC_KEY, signature, timestamp, rawBody)
if (!valid) return { statusCode: 401, body: 'Invalid request signature' }
```

Use `crypto.subtle` (Node 20 built-in — no extra package needed):

```typescript
async function verifyDiscordSignature(publicKey: string, signature: string, timestamp: string, rawBody: string): Promise<boolean> {
  const key = await crypto.subtle.importKey('raw', Buffer.from(publicKey, 'hex'), { name: 'Ed25519' }, false, ['verify'])
  return crypto.subtle.verify('Ed25519', key, Buffer.from(signature, 'hex'), new TextEncoder().encode(timestamp + rawBody))
}
```

**Interaction type 1 (PING):** Return `{ type: 1 }` immediately.

**Interaction type 2, command `setup`:**
1. Extract `guild_id` (Discord server ID) and `channel_id` from the interaction payload
2. Generate a random UUID token
3. Write `{ token, discordGuildId, channelId, ttl: now + 600 }` to the pending table
4. Return an ephemeral response (type 4, flags: 64) with the verification link:
   ```
   Click here to verify your WoW guild membership (expires in 10 minutes):
   https://{cloudfront-domain}/discord-auth?token={uuid}
   ```

**Interaction type 2, command `keys`:**
1. Extract `guild_id` from the interaction payload, look up config table
2. If no config: ephemeral message — "This server hasn't been set up yet. Run `/setup` to get started."
3. Extract options: `min-level` (int), `max-level` (int), `dungeon` (string)
4. If `dungeon` provided, resolve name → ID via the dungeon map (case-insensitive substring match). If no match, return ephemeral error listing valid dungeon names
5. Query keys table for `wowGuildId` with any level/dungeon filters
6. Format and return a Discord embed response (type 4)

**`/keys` response format:**
```json
{
  "type": 4,
  "data": {
    "embeds": [{
      "title": "Mythic+ Keys — Guild Name",
      "color": 16750848,
      "fields": [{ "name": "CharacterName", "value": "🟠 +15 Dungeon Name", "inline": true }],
      "timestamp": "...",
      "footer": { "text": "via Key Coord" }
    }],
    "flags": 64
  }
}
```

Key level emoji tiers: `🟠` ≥15, `🟣` 13–14, `🔵` 10–12, `🟢` 6–9, `⚪` <6. Sort descending by key level. If >25 keys, show top 25 + note to visit website. If 0 keys, show "No keys found" description.

**Env vars:** `DISCORD_CONFIG_TABLE_NAME`, `DISCORD_PENDING_TABLE_NAME`, `TABLE_NAME`, `DISCORD_PUBLIC_KEY`, `SITE_URL`

**Lambda timeout: 10s.** Discord's hard cutoff is 3s.

### `src/handlers/discordSetup.ts`

**Route:** `POST /discord/setup`

Called by the website's `/discord-auth` page after Blizzard OAuth completes. This is where guild membership is verified and the config is saved.

**Auth:** Blizzard Bearer token (same pattern as `putKey.ts`)

**Request body:**
```json
{ "token": "<uuid from pending table>" }
```

**Flow:**
1. Look up `token` in the pending table → 404 if not found, 410 if TTL expired
2. Extract `discordGuildId` and `channelId` from the pending record
3. Call Blizzard API `/profile/user/wow` to get authenticated user's characters
4. Parse `wowGuildId` from the pending record — wait, the pending record only has `discordGuildId` and `channelId`. The WoW guild must come from the user's Blizzard profile.

   **Determining `wowGuildId`:** Fetch the user's characters, find which guild the character belongs to. If the user has characters in multiple guilds, the character they authenticated as (via the existing character select flow) should determine the guild. The website should pass the selected character's guild info alongside the token call.

   **Revised request body:**
   ```json
   { "token": "<uuid>", "wowGuildId": "us-area-52-example-guild", "region": "us", "realmSlug": "area-52" }
   ```

5. Call Blizzard API `/data/wow/guild/{realm}/{guild}/roster?namespace=profile-{region}`
6. Verify at least one of the user's characters is in the guild roster → 403 if not
   - For now, any guild member may configure. Rank enforcement is TBD — the Blizzard API returns a numeric `rank` (0 = GM, 1–9 = other ranks), but rank names are guild-customizable and not exposed via the API.
7. Write config to `key-coord-discord-config-table`: `{ discordGuildId, sk: 'CONFIG', wowGuildId, channelId, configuredBy: battletag, configuredAt, notificationsEnabled: true }`
8. Delete the pending token record
9. Use the bot token to POST "✅ Setup complete! This server will now receive key update notifications for **{guild name}**." to `channelId` via Discord API: `POST https://discord.com/api/v10/channels/{channelId}/messages`
10. Return `{ success: true }`

**Bot token access:** Fetch from Secrets Manager. Cache in module scope (Lambda container reuse) to avoid a Secrets Manager round-trip on every warm invocation.

**Env vars:** `DISCORD_CONFIG_TABLE_NAME`, `DISCORD_PENDING_TABLE_NAME`, `DISCORD_SECRET_NAME`, `BLIZZARD_CLIENT_ID`

### `src/handlers/discordNotification.ts`

**Trigger:** DynamoDB Stream on the keys table (INSERT and MODIFY events only — REMOVE is filtered out at the event source level)

This Lambda has no HTTP route. It is invoked automatically when a key is written to DynamoDB.

**Event source configuration (in CDK):**
```typescript
import * as lambdaEventSources from 'aws-cdk-lib/aws-lambda-event-sources'

discordNotificationFn.addEventSource(new lambdaEventSources.DynamoEventSource(table, {
  startingPosition: lambda.StartingPosition.LATEST,
  batchSize: 1,
  filters: [
    lambda.FilterCriteria.filter({
      eventName: lambda.FilterRule.or('INSERT', 'MODIFY'),
    }),
  ],
}))
```

**Handler flow:**
1. Extract `NewImage` from the stream record — unmarshal `guildId`, `characterName`, `dungeonId`, `keyLevel`, `updatedAt`
2. Query `wowGuildId-index` GSI on the config table using `guildId`
3. If no config or `notificationsEnabled: false`, skip silently
4. Fetch bot token from Secrets Manager (cached in module scope across warm invocations)
5. Build embed payload and `POST https://discord.com/api/v10/channels/{channelId}/messages` with `Authorization: Bot {token}`
6. On 429: log the rate-limit, do not retry — the next key update will trigger a new invocation
7. On non-2xx: log the error but do not throw — throwing would cause Lambda to retry from the stream, risking duplicate notifications

**Notification embed:**
```json
{
  "embeds": [{
    "description": "**CharacterName** now holds a **+14 Pit of Saron**",
    "color": 16750848,
    "fields": [
      { "name": "Character", "value": "CharacterName", "inline": true },
      { "name": "Dungeon", "value": "Pit of Saron", "inline": true },
      { "name": "Level", "value": "+14", "inline": true }
    ],
    "timestamp": "...",
    "footer": { "text": "via Key Coord" }
  }]
}
```

No `@here` or `@everyone` mentions — reserved for the stretch-goal ping feature.

**Env vars:** `DISCORD_CONFIG_TABLE_NAME`, `DISCORD_SECRET_NAME`

### `website/src/pages/DiscordAuth.tsx` (new file)

New page rendered at `/discord-auth`. Handles the post-OAuth step of the Discord setup flow.

**Flow:**
1. On mount, read `?token=` from the URL
2. If user is not logged in with Blizzard → store token in `sessionStorage` → trigger Blizzard OAuth (same as existing login flow). After OAuth callback, `AuthContext` should redirect back here with the token.
3. If user is logged in but has no selected character → show character select (same as main app flow)
4. Once logged in with a selected character → call `POST /api/discord/setup` with `{ token, wowGuildId, region, realmSlug }` and the Blizzard Bearer token
5. Show result:
   - Success: "✅ You're all set! Your Discord server will now receive key update notifications for **{guild name}**. You can close this tab."
   - 403: "You don't appear to be a member of that guild."
   - 404/410: "This setup link has expired. Run `/setup` in Discord again to get a new one."
   - Other error: "Something went wrong. Please try again."

**Route:** Add `/discord-auth` to `main.tsx` (or wherever routing is defined).

## Discord Application Setup (One-time, Manual)

1. Create a Discord Application at discord.com/developers
2. Note **Application ID** and **Public Key** from General Information
3. Create a Bot; save credentials in AWS Secrets Manager:
   ```bash
   aws secretsmanager create-secret --name "key-coord/discord-prod" \
     --secret-string '{"publicKey":"...","botToken":"...","applicationId":"..."}'
   ```
4. Register slash commands:
   ```bash
   # /setup command
   curl -X POST "https://discord.com/api/v10/applications/{APP_ID}/commands" \
     -H "Authorization: Bot {BOT_TOKEN}" -H "Content-Type: application/json" \
     -d '{"name":"setup","type":1,"description":"Connect this Discord server to your WoW guild for key notifications"}'

   # /keys command
   curl -X POST "https://discord.com/api/v10/applications/{APP_ID}/commands" \
     -H "Authorization: Bot {BOT_TOKEN}" -H "Content-Type: application/json" \
     -d '{"name":"keys","type":1,"description":"Show Mythic+ keys for this server'\''s WoW guild","options":[{"name":"min-level","description":"Minimum key level","type":4,"required":false},{"name":"max-level","description":"Maximum key level","type":4,"required":false},{"name":"dungeon","description":"Filter by dungeon name","type":3,"required":false}]}'
   ```
5. Set **Interactions Endpoint URL** in Developer Portal: `https://{cloudfront-domain}/api/discord/interactions`
   - Discord will verify with a PING immediately — Lambda must respond `{ type: 1 }` within ~3s
6. Generate bot invite URL with **both** `bot` and `applications.commands` scopes. The bot scope needs the `Send Messages` permission so it can post to channels.

**Deploy before step 5**: The Lambda and route must exist before Discord can verify the endpoint.

- [ ] Click link → Blizzard OAuth → redirected to `/discord-auth`
- [ ] Completing auth as a guild member → `POST /discord/setup` returns 200, config saved to DynamoDB, bot posts "✅ Setup complete!" in the channel
- [ ] Completing auth as a non-guild-member → 403 shown on the page
- [ ] Token expires after 10 minutes → 410 shown on the page
- [ ] Running `/setup` twice before completing → second token overwrites first in pending table (or both exist; either is fine since both point to same server)

### `/keys` Slash Command
- [ ] Server with no configured guild → ephemeral "run /setup to get started" message
- [ ] Configured server → embed with keys sorted by level descending
- [ ] `min-level:10` filter works
- [ ] `dungeon:pit of saron` (case-insensitive) filter works
- [ ] Unknown dungeon name → ephemeral error with valid dungeon list
- [ ] 0 keys → "No keys found" message

### Key Update Notification (Stream-triggered)
- [ ] Submit a key → stream fires → `discordNotification` Lambda invoked → message appears in Discord channel within ~10s
- [ ] Delete a key → no Discord notification (REMOVE filtered at event source)
- [ ] Guild with no Discord config → Lambda exits silently (confirm in CloudWatch)
- [ ] `notificationsEnabled: false` → Lambda exits silently
- [ ] Bot token cached: second invocation in same Lambda container does not call Secrets Manager again

### End-to-End
- [ ] Run `/setup` → complete Blizzard OAuth → submit key via addon → notification appears in Discord → `/keys` shows the key
